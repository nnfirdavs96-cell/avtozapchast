# План: универсальный каталог запчастей по VIN (как у autodoc.ru)

> Статус: **проект плана, к разработке не приступали.** Документ описывает целевую
> архитектуру и этапы. Старт — по отдельной команде владельца.

## 0. Цель одной фразой

Заказчик покупает подписку у **любого** сервиса-каталога → вставляет ключ в
админке → нажимает «Проверить» (🟢) → «Включить» → **каталог по VIN работает как у
autodoc.ru**. Без программиста и без правок кода при смене провайдера.

---

## 1. Принцип autodoc.ru, который повторяем

Два независимых слоя, склеенные **OEM-номером** (артикул + производитель):

```
            ┌──────────────── КАТАЛОГ ────────────────┐   ┌────── ЦЕНЫ ──────┐
VIN/параметры → авто → дерево узлов → деталь + OEM-номер  →  цена/наличие → корзина
            └── CatalogProvider (адаптер/профиль) ──┘        └─ PriceProvider ─┘
                                          мост = OEM-номер
```

- **Каталог** знает машины (VIN → авто → узлы → детали с OEM + аналоги), цен не знает.
- **Цены** знают деньги (OEM → цена/наличие), машин не знают.
- У autodoc.ru: каталог = Laximo, цены = свой склад. У нас то же, но провайдер сменный.

---

## 2. Ядро универсальности: адаптер, управляемый данными (профилем)

Чтобы смена сервиса была **настройкой, а не кодом**, провайдер описывается
**профилем** (данные в БД, правится в админке), а не отдельным PHP-классом.

### 2.1. Профиль провайдера (пример UMAPI)

```jsonc
{
  "id": "umapi",
  "title": "UMAPI",
  "transport": "rest",                 // rest | soap | adapter(код)
  "base_url": "https://api.umapi.ru/",
  "auth": { "type": "query", "param": "key" },   // query | header | bearer
  "credentials": ["key"],              // какие поля показать в админке
  "endpoints": {
    "searchVin": "?method=getVehicle&vin={VIN}&key={KEY}",
    "getGroups": "?method=getGroups&ssd={SSD}&key={KEY}",
    "getParts":  "?method=getParts&group={GROUP}&key={KEY}",
    "getCrosses":"?method=getCrosses&oem={OEM}&key={KEY}",
    "getPrice":  "?method=getPrice&oem={OEM}&key={KEY}"
  },
  "map": {                             // где в ответе что лежит (JSONPath)
    "vehicle.make":   "data.brand",
    "vehicle.model":  "data.model",
    "vehicle.year":   "data.year",
    "groups[].id":    "items[*].id",
    "groups[].name":  "items[*].name",
    "parts[].oem":    "items[*].article",
    "parts[].brand":  "items[*].manufacturer",
    "parts[].name":   "items[*].name"
  },
  "capabilities": { "vin": true, "schemas": false, "crosses": true }
}
```

### 2.2. Один класс на все REST-сервисы

`GenericRestAdapter` берёт профиль → подставляет `{VIN}/{KEY}/{OEM}/…` в шаблоны →
ходит через наш `httpGet()` → парсит ответ по `map` → отдаёт **наш единый формат**.
Фронт/корзина/цены не знают, какой сервис под капотом.

**Итог:** новый REST-сервис = новый профиль (форма в админке или готовый JSON),
**без единой строки PHP**.

### 2.3. Экзотика → код-адаптер (пишем заранее)

Сервисы, которые профилем не покрыть (нестандартный протокол), реализуют тот же
интерфейс отдельным классом. На рынке автозапчастей это практически только:

- **Laximo** — SOAP + цепочка токенов `ssd`/`carSsd` между шагами, авторизация
  логин+пароль. Профилем не описывается → пишем `LaximoAdapter` **заранее, в
  рамках проекта** (обязательный пункт, не «резерв»).

Для клиента Laximo в админке выглядит так же, как все: выбрал → ввёл логин/пароль →
проверил → включил. Разница только внутренняя (адаптер уже в коде к моменту покупки).

## 3. Единые интерфейсы (контракт для всех провайдеров)

```php
interface CatalogProvider {
    public function id(): string;
    public function searchVin(string $vin): ?array;               // авто по VIN
    public function getBrands(): array;
    public function getModels(string $brandId): array;
    public function getModifications(string $modelId): array;     // годы/двигатели
    public function getGroups(array $vehicleCtx): array;          // дерево узлов
    public function getParts(array $vehicleCtx, $groupId): array; // детали + OEM
    public function getCrosses(string $oem, string $brand = ''): array;
    public function capabilities(): array;
    public function testConnection(): array;   // для кнопки «Проверить»
}

interface PriceProvider {
    public function priceByOem(string $oem, string $brand = ''): ?array; // цена/наличие/url
}
```

Фабрики:
- `CatalogProviderFactory::current()` — читает настройку `catalog_provider`, по
  профилю отдаёт `GenericRestAdapter`, по коду — `LaximoAdapter`/`PartsApiAdapter`/`MockAdapter`.
- `PriceAggregator` — `WarehousePriceProvider` (свой склад) → `AutoEuroPriceProvider`
  (фолбэк) → цена в **сомони** с наценкой.

## 4. Админка — «активация после покупки»

Экран **Суперадмин → Каталог** (расширяем существующий блок в `superadmin/vin.php`):

| Элемент | Поведение |
|---------|-----------|
| **Провайдер** (выпадающий) | список профилей + код-адаптеров (Демо, PartsAPI, UMAPI, Laximo, …) |
| **Поля доступа** | показываются по `credentials` профиля (1 ключ / логин+пароль / …) |
| **«Проверить соединение»** | живой `testConnection()` → 🟢/🔴 + пример ответа |
| **«Включить каталог»** | тумблер, активен когда провайдер выбран и тест 🟢 |
| **«+ Новый провайдер»** | форма профиля для сервиса, которого нет в списке (без кода) |
| **Слой цен** | свой склад / + AutoEuro (вкл/выкл фолбэк) |

**Сценарий заказчика (любой провайдер, 3 шага, без программиста):**
1. Купил подписку/ключ.
2. Админка → выбрал провайдера → ввёл доступы → «Проверить» → 🟢.
3. «Включить» → VIN-поиск, дерево узлов, детали, цены в сомони, корзина — работают.

## 5. Что переиспользуем (уже в репозитории, ~60% готово)

| Уже есть | Роль в новой схеме |
|----------|--------------------|
| `pages/vin.php` (PR #196) | фронт: табы, дерево узлов, кроссы, карточка авто |
| `CatalogApi` (`searchByVinCat`/`getCrosses`/`enrichFromWarehouse`) | → `PartsApiAdapter` + основа `PriceProvider` |
| `VinService` (декодер VIN + кэш) | fallback-декодер, когда у провайдера нет VIN |
| `httpGet()` + обработка лимита `error_code 5000` | транспорт всех REST-адаптеров |
| `AutoEuro` (`searchItems`/`createOrder`/`fromSettings`) | `AutoEuroPriceProvider` почти даром |
| `partsapi_kv_cache` / `partsapi_catalog_cache` | → общий `catalog_cache` (ключ по провайдеру) |
| Админ-настройки `catalog_api_*` | расширяем: `catalog_provider` + профили + доступы |

## 6. Изменения данных (миграции — идемпотентные, как принято в проекте)

- `site_settings`: `catalog_provider` (id активного), `catalog_profiles` (JSON
  профилей), доступы хранить отдельными ключами с провайдер-префиксом
  (`cat_cred_umapi_key`, `cat_cred_laximo_login`, …).
- Таблица `catalog_cache` (общая): `cache_key` (провайдер+метод+аргументы),
  `result`, `cached_at`. Версионирование как сейчас (`CACHE_VER`).
- Корзина: `orders`/сессия — поддержка гостевого заказа (этап 4).

## 7. Этапы (каждый — отдельный PR, сайт рабочий после любого)

| Этап | Что | Риск | Эффект |
|------|-----|------|--------|
| **1. Каркас** | `CatalogProvider`+`PriceProvider`, `Factory`, `MockAdapter`, рефактор PartsAPI→`PartsApiAdapter`. Админка: выпадающий «Провайдер». Поведение сайта не меняется. | низкий | демо-каталог без ключа; фундамент |
| **2. Generic + профили** | `GenericRestAdapter` + хранилище профилей + форма «Новый провайдер» в админке. PartsAPI/UMAPI переводим на профили. | средний | смена REST-сервиса = настройка |
| **3. Слой цен** | `PriceAggregator` (склад + AutoEuro) → цены в сомони на деталях каталога. | низкий | реальные цены на находках |
| **4. Laximo** | `LaximoAdapter` (SOAP + ssd-токены, логин/пароль) — **обязательный**. | средний | оригинальные OEM-схемы |
| **5. Гостевая корзина** | снять login-wall, заказ по телефону без входа. | средний | заказ без регистрации |

## 8. Что нужно от владельца, чтобы «после покупки заработало само»

1. Стартовый провайдер для проверки каркаса (презентация рекомендует **UMAPI**).
2. **Тест-ключ UMAPI** (бесплатный тест-месяц) — проверить адаптер до боевой покупки.
3. Подтвердить слой цен: только свой склад или + AutoEuro (нужен ключ AutoEuro).
4. Если планируется Laximo — доступ к тест-аккаунту Laximo для проверки адаптера.

---

## 9. Честная граница (чтобы без сюрпризов)

- REST/JSON сервисы (UMAPI, PartsAPI, TecDoc, acat) — **полностью настройка из
  админки, без кода.**
- Laximo и подобная экзотика — **разовый код-адаптер**, который мы пишем заранее;
  после этого для клиента он в списке наравне со всеми.
- «Мгновенно любой, заранее не предусмотренный, не-REST сервис» — невозможно ни у
  кого (включая autodoc.ru). Наша схема покрывает все реальные сервисы рынка.

---

## 10. Резюме

Архитектура = **единый интерфейс + сменные провайдеры**, где:
- 95% сервисов подключаются **профилем-данными из админки без кода**;
- редкая экзотика (Laximo) — разовый адаптер, написанный заранее;
- слой цен (склад + AutoEuro, сомони) независим от каталога;
- фронт и корзина не меняются при смене провайдера.

Это и есть «как у autodoc.ru, но не привязано к одному поставщику и управляется из
админки».
